僕のローカルディレクトリ
一撃構築プロンプト
（ヒアリング型メタプロンプト）

僕が事業を回している「AI経営ローカルディレクトリ」──事業部フォルダ・ナレッジ3分類・ルールファイル体制・スキル管理・24時間ナレッジパイプライン──の設計思想を丸ごと移植するメタプロンプトです。人によって事業部・保有API・使用SaaSは違うので、AIがあなたにヒアリングしてから、あなた専用に最適化した構造を一撃で構築します。

1. 「ローカルディレクトリ」が経営のOSになる理由

ChatGPTやClaudeのチャット画面だけでAIを使っていると、毎回ゼロから文脈を説明することになります。僕がやっているのはその逆で、PCの中に「会社のフォルダ構造」を作り、そこにナレッジ・ルール・過去の成果物を全部貯めて、ローカルを読めるAI（Claude Code / Cursor / Codex）に参照させる方式です。

  • AIが過去の自分の文章・実績・商品情報を参照してアウトプットするので、出力が「自分のもの」になる
  • フォルダ構造とルールファイルがAIへの「就業規則」になるので、どのAIを使っても同じ品質で動く
  • 録音・会議・SaaSのデータが毎日ナレッジとして貯まる仕組み（ナレッジパイプライン）を作れば、使うほどAIが自分に詳しくなる

2. 移植する設計思想（僕の環境の全体像）

このメタプロンプトが移植するのは「フォルダの見た目」ではなく9つの設計思想です。ここを理解してから使うと、構築後のカスタマイズが自在になります。

#思想中身
1事業部ドリブンすべての成果物は「事業部フォルダ」のどこかに属する。事業部＝ミニ会社として独立完結させる（例：SNS事業部／プロダクト事業部／セールス／リストマーケ事業部／経理／経営企画／AI活用事業部）
2単一情報源1つの情報の実体は1箇所だけ。コピーを散らかさず、参照はリンクで増やす
3ナレッジ3分類横断ナレッジは「パーソナル（自分の人格・経験）／プロダクト（商品設計）／コンテンツ（文体・型）」の3分類＋「コンテンツ元本（SaaSからの未加工データ）」で管理
4directory.mdが正本「何をどこに保存するか」の判断マップを1ファイルに集約し、AIは保存前に必ずこれを読む。構造を変える時はこのファイルを先に更新する
5ルールは薄く・数本だけAIが毎回読むルールファイルは肥大化させない。憲法1本＋テーマ別数本に統合し、詳細は参照用フォルダに逃がす
6スキル＝定型作業の手順書繰り返す作業（リサーチ・台本・配信文など）はSKILL.md（手順書）にして事業部フォルダに置く。フィードバックはその場でSKILL.mdに反映し、使うほど賢くする
7タスク正本は1箇所タスク管理ツール（Trello等）を正本と決め、フォルダ内のタスクファイルは参照ビューに留める。二重管理をしない
824時間ナレッジパイプライン録音（Plaud）・会議（Zoom）・各SaaSのデータを毎日自動回収→文字起こし→「コンテンツ元本」へ格納→ナレッジ化。寝ている間も一次情報が貯まる
9AIでも人間でも迷わない構造フォルダごとに_INDEX.md（案内板）、ファイル名はYYYY-MM-DD_種類_テーマ、絵文字フォルダ名禁止（同期ツールが壊れる）、プライベートは専用フォルダに分離
🧠

いちばん大事な思想は④です。AIに「保存する前にdirectory.mdを読め」という一点を守らせるだけで、フォルダはずっと散らかりません。人間の会社で言う「決裁ルートの明文化」と同じで、構造よりも「構造を守らせる仕組み」の方が重要です。

3. 対応しているAI環境

このメタプロンプトは、ローカルのフォルダを読み書きできるAIならどれでも使えます。ヒアリングの中で使用AIを聞き、それに合わせた入口ファイル（CLAUDE.md / AGENTS.md等）を自動で作り分けます。

AI環境入口ファイル備考
Claude Code（デスクトップ/CLI）CLAUDE.md ＋ .claude/rules/ルールの自動読み込みが強力。いちばん推奨
CursorAGENTS.md（または .cursor/rules/）エディタ中心の人向け
Codex CLIAGENTS.mdChatGPTサブスク枠で動く
OpenCoder等のOSS系AGENTS.md（READMEにルール記載）読み込み仕様に合わせて調整
サブエージェント運用上記＋エージェント定義事業部ごとに担当AI（ペルソナ）を割り当てる拡張も可能

4. 一撃構築メタプロンプト本体

以下をコピーして、Claude Code（またはCursor / Codex）に貼り付けて送信してください。AIがまずあなたの事業・ツール・運用スタイルをヒアリングし、設計図の承認を取ってから、フォルダ・ルールファイル・運用ドキュメントを一括構築します。所要時間は約15〜30分（ヒアリング込み）です。

星川式ローカルディレクトリ 一撃構築メタプロンプト
あなたは「AI経営ローカルディレクトリ」構築の専門エージェントです。
これから、このPCの中に「AIと一緒に事業を回すためのローカルディレクトリ」
（フォルダ構造＋ルールファイル＋運用ドキュメント一式）を構築してください。

これは星川式のディレクトリ設計思想を、私の事業内容に合わせて移植するプロジェクトです。
最初に必ず下の【設計思想】を読み込み、全工程でこの思想に従ってください。

■ 設計思想（全工程で厳守）
━━━━━━━━━━━━━━━━━━━━━━
1. 事業部ドリブン：すべての成果物は「事業部フォルダ」のどこかに属する。
   事業部＝ミニ会社。個別案件・成果物は事業部に、横断ナレッジは共通ナレッジに入れる。
2. 単一情報源：1つの情報の実体は1箇所だけ。重複保存禁止。参照はINDEXへのリンクで行う。
3. ナレッジ3分類＋元本：
   - パーソナルナレッジ＝本人の人格・価値観・経験・実績
   - プロダクトナレッジ＝商品設計・ペルソナ・差別化
   - コンテンツナレッジ＝文体・口調・過去コンテンツの型・フィードバック集
   - コンテンツ元本＝SaaS・録音などの未加工一次データ（加工前の原文を消さない）
4. directory.md が出力先判断の正本：AIは何かを保存する前に必ず directory.md を読む。
   構造を変える時は directory.md を先に（または同時に）更新する。
5. ルールは薄く・数本だけ：毎回自動で読むルールは「憲法1本＋テーマ別3本」まで。
   詳細・履歴は参照用フォルダ（docs/）に逃がし、必要な時だけ読む。
6. スキル＝定型作業の手順書：繰り返し作業は SKILL.md にして該当事業部の スキル/ に置く。
   実行前に全文を読み、ユーザーのフィードバックはその場で SKILL.md に反映する。
7. タスク正本は1箇所：ユーザーが使うタスク管理ツールを正本とし、
   フォルダ内の タスク.md は参照ビューに留める（二重管理禁止）。
8. 24時間ナレッジパイプライン：録音・会議・SaaSデータを
   「受信箱 → コンテンツ元本 → ナレッジ」の3層で流す。自動化は任意、まず手動手順を確立する。
9. AIでも人間でも迷わない構造：
   - 各フォルダに _INDEX.md（目的1行＋サブフォルダ表＋使い方）
   - ファイル名は YYYY-MM-DD_種類_テーマ.md
   - フォルダ名への絵文字使用は禁止（同期ツールが壊れる事故の原因）
   - プライベート情報は専用フォルダに分離し、仕事の成果物に混ぜない

■ STEP 0：ヒアリング（1問ずつ・全11問）
━━━━━━━━━━━━━━━━━━━━━━
【A. 事業について】
Q1. あなたの事業内容を教えてください（複数OK。例：講座販売／コンテンツ販売／SNS運用代行／受託制作／コーチング・コンサル／物販／会社員の副業）
Q2. 収益の柱になっている（または作りたい）商品・サービスは何ですか？
Q3. 使っている発信媒体はどれですか？（YouTube／X／Instagram／TikTok／note／メルマガ／LINE／ブログ／ポッドキャスト等）
Q4. チームで動いていますか、個人ですか？（外注さんの有無も）

【B. ツールについて】
Q5. 使用するAIエディタは？（Claude Code／Cursor／Codex CLI／その他。複数併用も可）
Q6. 日常的に使っているSaaS・ツールは？（Google Workspace／Notion／Slack／Chatwork／Discord／Zoom／Plaud等の録音ツール／freee等の会計ソフト、など思いつくだけ）
Q7. すでに持っているAPIキー・つなぎたいAPIやMCPはありますか？（無ければ「無し」でOK。後から追加できます）
Q8. タスク管理は何を使っていますか？（Trello／Notion／Todoist／Googleカレンダーだけ／決めていない）

【C. 運用について】
Q9. プライベートの情報（健康・家族・個人の内省メモ等）も同じフォルダ体系で管理したいですか？（管理する場合は専用フォルダに分離します）
Q10. 複数のPCで同期しますか？（する場合：Syncthing／iCloud／Dropbox等。同期ツールがある場合、絵文字フォルダ名禁止などの制約を強めに適用します）
Q11. ルートフォルダの名前と置き場所の希望はありますか？（例：~/AIフォルダ_屋号。無ければ「AIフォルダ_{屋号または名前}」をホーム直下に提案します）

■ STEP 1：設計図の提案（承認ゲート）
━━━━━━━━━━━━━━━━━━━━━━
ヒアリング結果から、以下の要領でフォルダツリー全体を設計し、ツリー図で提示して承認を得ること。

【事業部フォルダの設計基準】
- Q1〜Q3の回答から、その人の事業を3〜7個の事業部に分ける。参考の型：
  - SNS事業部（発信・集客。媒体ごとのサブフォルダ：企画／台本／投稿／分析／リサーチ／ナレッジ／スキル）
  - プロダクト事業部（商品・顧客対応。商品ごとのサブフォルダ：商品概要／教材／顧客サポート／販売導線）
  - セールス（セミナー／個別相談／LP・セールスライティング）
  - リストマーケ事業部（LINE／メルマガ／ローンチ）
  - 経理（会計・請求・契約。法人と個人事業が両方ある場合は分ける）
  - 経営企画（アイディア・新商品企画・KPI・プロジェクト）
  - AI活用事業部（AI環境そのものの改善・スキル管理・自動化スクリプト）
- 該当しない事業部は作らない。逆にQ1で特殊な事業（例：物販、受託）がある場合は専用事業部を立てる。
- 各事業部の直下は「タスク.md（参照ビュー）＋事業部管理/＋機能サブフォルダ＋スキル/」を基本形にする。

【共通構造（全員共通で作る）】
{ルート}/
├── directory.md            ← 出力先判断の正本（STEP 3で生成）
├── {入口ファイル}           ← CLAUDE.md / AGENTS.md（Q5の使用AIに合わせて）
├── タスク管理ルール.md
├── 事業部/
│   └── {設計した各事業部}/
├── ナレッジ/
│   ├── 共通ナレッジ/
│   │   ├── パーソナルナレッジ/
│   │   ├── プロダクトナレッジ/
│   │   ├── コンテンツナレッジ/
│   │   ├── タスク攻略ナレッジ/   ← How-to・理論・プロンプト集
│   │   └── AI設定/              ← APIキーの置き場所メモ等（キー本体は.env）
│   └── コンテンツ元本/
│       ├── _受信箱/             ← パイプラインの入口（未処理データ）
│       └── {Q6のSaaSごとのフォルダ}/（例：Plaud/ Zoom/ Google_Docs/ LINE/）
├── アーカイブ/
│   └── 削除対象_手動確認/       ← AIは直接削除せず、ここに集めて人間が消す
└── .claude/ または .agents/     ← ルール・スクリプト置き場（Q5に合わせる）
    ├── rules/（4本）
    └── scripts/

【プライベート分離（Q9がYESの場合）】
- ナレッジ/共通ナレッジ/タスク攻略ナレッジ/プライベート/ を作り、
  「プライベートは仕事の成果物・配信文・外部送信に混ぜない」ルールを憲法に明記する。

■ STEP 2：フォルダ実体の作成
━━━━━━━━━━━━━━━━━━━━━━
承認された設計図どおりにフォルダを一括作成する。
- すべてのフォルダに _INDEX.md を同時作成（テンプレ：目的1行／中身の表／使い方2行）
- フォルダ名に絵文字を使わない
- 空フォルダには .gitkeep 等は不要（_INDEX.mdがあるため）

━━━━━━━━━━━━━━━━━━━━━━
■ STEP 3：ルールファイルの生成
━━━━━━━━━━━━━━━━━━━━━━
以下の5ファイルを、ヒアリング内容を反映した「その人専用」の内容で書き切ること。
テンプレの丸写しではなく、事業部名・商品名・SaaS名を実際の回答で埋めること。

(1) directory.md（最重要）
   - 冒頭に「AIは何かを保存する前に必ずこのファイルを読む」と明記
   - 設計思想（上の9箇条をその人向けに要約）
   - フォルダ構造の全体マップ（ツリー図）
   - 出力先判断フロー：
     「保存したい → プライベートか？ → 横断ナレッジか？ → どの事業部の責任範囲か？ → 該当フォルダへ YYYY-MM-DD_種類_テーマ.md で保存」
   - 事業部ごとの判断キーワード表（例：LINE配信・メルマガ→リストマーケ事業部）
   - 新フォルダを作る時の手順（_INDEX.md同時作成＋親INDEXへの追記＋directory.md更新）

(2) 入口ファイル（CLAUDE.md／AGENTS.md：Q5の使用AIに合わせて。併用なら両方作り内容は共通化）
   - 「意思決定は directory.md を正本として行う」
   - 最優先ルール5行（保存前にdirectory.md／削除・外部送信・公開は確認必須／APIキーは.env／ルールファイルの場所／タスク正本）

(3) rules/00_運用憲法.md
   - 絶対ゲート：削除・外部送信・公開・課金が発生する操作は、ユーザーの明示同意を得てから実行
   - APIキー・トークン・.envの中身を表示・保存・出力しない
   - 推測禁止：パス・数値・フォーマットは実物を読んでから使う
   - 保存判断の順序（directory.md参照）
   - タスクと記録の正本の場所

(4) rules/01_保存と命名.md
   - 情報は原則.md／ファイル名 YYYY-MM-DD_種類_テーマ／改訂は_v2
   - 単一フォルダ所属・重複保存禁止
   - 新フォルダの4点セット（作成＋_INDEX.md＋親INDEX追記＋directory.md反映）
   - 絵文字フォルダ名禁止／_未分類フォルダへの放り込み禁止

(5) rules/02_コンテンツ制作.md
   - 一次情報ドリブン：読者に届く文章は、本人の過去の文章・実績・語彙を参照して作る
   - AIが表現・実績数字・エピソードをゼロから創作しない。
     出典のない数字・強い主張は「[要確認]」と書いて残す
   - 生成物の末尾に「参照した元ファイル」を出典として記録する
   ※ Q5でClaude Code使用なら .claude/rules/ に、それ以外なら {ルート}/rules/ に置き、
     入口ファイルから参照させる

━━━━━━━━━━━━━━━━━━━━━━
■ STEP 4：ナレッジパイプラインの設計
━━━━━━━━━━━━━━━━━━━━━━
Q6で挙がったSaaS・録音ツールに合わせて、「ナレッジが毎日貯まる流れ」を作る。

(1) まず手動版の手順書を作る：
   ナレッジ/コンテンツ元本/_受信箱/README.md に
   「データの入れ方 → AIに『受信箱を処理して』と言う → AIが元本フォルダへ振り分け＋要約ナレッジ化」
   の手順を書く。
(2) 「受信箱を処理して」と言われた時のAIの動作手順を、
   .claude/skills/（またはskills/）knowledge-pipeline/SKILL.md として作成：
   - _受信箱のファイルを列挙 → 種類判定（録音/議事録/書き出しデータ/メモ）
   - 音声はユーザーの文字起こし手段を確認（ローカルwhisper／Groq API／手動）
   - 元本を コンテンツ元本/{SaaS名}/ へ移動し、要約を 共通ナレッジ/ の該当分類へ.mdで保存
   - 元本と要約を相互リンク
(3) 自動化を希望された場合のみ、OS標準のスケジューラ（Mac: launchd／Windows: タスクスケジューラ）
   またはAIエディタの定期実行機能で日次実行する案を提示する（実装は任意）。

━━━━━━━━━━━━━━━━━━━━━━
■ STEP 5：スキルのひな形を2本作る
━━━━━━━━━━━━━━━━━━━━━━
その人の事業に最も関係が深い定型作業を2つ選び（例：SNS投稿作成／リサーチシート作成／配信文作成）、
該当事業部の スキル/{スキル名}/SKILL.md として以下の骨組みで作成：
- 発動条件（どんな依頼で使うか）
- 実行前に読むファイル（本人の過去成果物・ナレッジのパス）
- 手順（ステップ形式）
- 出力先とファイル名規則
- フィードバック反映欄（ユーザーの修正指示をここに追記していく）

━━━━━━━━━━━━━━━━━━━━━━
■ STEP 6：セキュリティと同期の設定
━━━━━━━━━━━━━━━━━━━━━━
- ルート直下に .env.example（KEY名だけのテンプレ）と .gitignore（.env／機密フォルダを除外）を作成
- APIキーの実体は .env に。Q7で挙がったAPIのKEY名をテンプレに列挙しておく
- Q10で同期ツール使用の場合：絵文字禁止の再確認、同期対象から .env を除外する設定を案内
- 「キーをチャットに貼らない・キーが漏れたら再発行」の2原則を README に明記

━━━━━━━━━━━━━━━━━━━━━━
■ STEP 7：検収テストと使い始めガイド
━━━━━━━━━━━━━━━━━━━━━━
(1) 検収テストを実演する：
   - テスト1：「YouTubeの企画メモを保存したい。どこに保存する？」と自問し、
     directory.mdの判断フローに従った回答を提示（正しい保存先が返るか）
   - テスト2：ダミーのメモを実際に正しいフォルダにYYYY-MM-DD形式で保存してみせる
   - テスト3：_受信箱にダミーファイルを置き、パイプラインを1周回してみせる
(2) README.md（使い始めガイド）をルートに作成：
   - 毎日の使い方3行（①データは受信箱へ ②作業はAIエディタで「〜して」 ③保存先はAI任せでOK）
   - 週1メンテ（受信箱ゼロ確認・変な場所のファイルをAIに再配置させる）
   - 構造を変えたくなったら directory.md を先に更新する
(3) 最後に、構築した全ファイルの一覧と、次にやるべきこと（特典①のリサーチ環境構築や
   特典②のAPI/MCP導入など）を3行で提案して完了。

━━━━━━━━━━━━━━━━━━━━━━
■ 禁止事項（全工程）
━━━━━━━━━━━━━━━━━━━━━━
- ユーザーの承認なしに既存ファイルを移動・削除・上書きしない
- フォルダ名に絵文字を使わない
- ルールファイルを5本より増やさない（肥大化はAIの読み落としの原因）
- APIキー等の秘密情報をチャット・ファイルに平文で出力しない（.envのみ）
- ヒアリングを飛ばして推測で構築しない
- 「_未分類」「misc」のような逃げ場フォルダを作らない

それでは、STEP 0 の Q1 から始めてください。

5. 構築後の運用のコツ（僕がやっていること）

最初の1週間は「保存先どこ？」をAIに聞きまくる

構築直後は、何かを保存するたびに「これどこに保存する？」とAIに聞いてください。directory.mdの判断フローがあなたの実態とズレている箇所が見つかるので、ズレたらdirectory.mdを直す。この往復が構造を「自分のもの」にします。

フィードバックは口で言って、ルールに書かせる

AIの出力に「もっとこうして」と思ったら、その場で伝えて該当のSKILL.mdやルールファイルに追記までさせること。会話の中だけで直させると、次のセッションで元に戻ります。ファイルに書かれた指示だけが資産になります。

ナレッジパイプラインは「手動で回る」ようになってから自動化する

いきなり全自動を目指すと壊れた時に直せません。まず「受信箱に入れる→『処理して』と言う」の手動運用を2週間回し、流れが固まってから定期実行に乗せるのが安全です。

月1回、AIに「構造の健康診断」をさせる

「ルール違反のファイル名・置き場所がおかしいファイル・_INDEX.mdが無いフォルダを洗い出して」と月1で依頼すると、散らかりが自動で見つかります。直すのもAIにやらせます（削除だけは自分で確認）。
